# 轻量级智能体设计讨论

## 1. 关于 Step 类型扩展

除了 `tool` 和 `llm` 这两种基本类型，为了让轻量级智能体能够处理更复杂的逻辑和任务流，我建议可以添加以下几种 Step 类型：

1.  **`tool` (工具调用)**
    *   **用途**：这是与外部世界交互的核心。用于调用在注册中心注册的外部技能、自定义函数或第三方 API。
    *   **示例**：`{"type": "tool", "action": "web_search", "args": {"query": "最新 AI 趋势"}}`

2.  **`llm` (LLM 直接生成/推理)**
    *   **用途**：直接利用大型语言模型（LLM）的文本生成和推理能力，进行文本处理、总结、创意写作、或简单的问答。
    *   **示例**：`{"type": "llm", "action": "summarize_text", "args": {"text": "${prev_step_output}", "length": "short"}}`

3.  **`conditional` (条件判断)**
    *   **用途**：引入流程控制。根据前一步骤的执行结果、某个变量的值或特定条件，决定接下来执行哪个分支的步骤序列。
    *   **示例**：
        ```json
        {
          "type": "conditional",
          "condition": "if ${prev_step_output.success} == true",
          "then": [
            { "type": "llm", "action": "generate_success_message", "args": {} }
          ],
          "else": [
            { "type": "llm", "action": "generate_error_message", "args": {} }
          ]
        }
        ```

4.  **`loop` (循环执行)**
    *   **用途**：实现重复操作。可以对一个列表的每个元素重复执行一系列步骤，或者重复某个操作直到满足特定的终止条件。
    *   **示例**：
        ```json
        {
          "type": "loop",
          "iterate_over": "${data_list_from_prev_step}",
          "loop_var": "item",
          "steps": [
            { "type": "tool", "action": "process_item", "args": {"input": "${item}"} }
          ]
        }
        ```

5.  **`user_input` (用户交互)**
    *   **用途**：在智能体执行任务的特定阶段，需要暂停并向用户提问、获取用户确认、或收集额外的必要信息。
    *   **示例**：`{"type": "user_input", "question": "请问您需要我生成报告的哪种格式？", "options": ["Markdown", "PDF"]}`

6.  **`store_state` (存储状态/变量)**
    *   **用途**：显式地将某个中间结果或特定值存储到一个命名变量中，供后续任意步骤引用。这有助于管理任务执行过程中的关键数据。
    *   **示例**：`{"type": "store_state", "key": "research_topic", "value": "${llm_output_topic_extraction}"}`

**总结**：`conditional` 和 `loop` 是最常用的控制流类型，可以作为初期实现的核心扩展。其他类型可以根据实际需求逐步添加。

## 2. 关于简单问候与快速回答模式

您的想法非常切中要害，**绝对应该有一个快速回答模式！** 让 LLM 直接处理简单的问候或问答，而不是每次都启动完整的规划-执行流程，是提高效率和节约 Token 的关键。

我建议的方案是：**前端/入口智能分类器 (Router/Classifier) + 快速回答模式 + 规划模式 + 用户干预机制。**

### 2.1. 智能分类器 (Router/Classifier)

*   **位置**：在接收到用户输入后，作为系统的前端或入口层。
*   **功能**：通过一个**轻量级、Token 效率高**的 LLM 调用（或者甚至是一些基于规则的判断），对用户意图进行初步分类。
*   **分类标准**：
    *   **闲聊/问候/简单问答**：用户意图明确，可以直接由 LLM 回答，无需外部工具或复杂规划。例如：“你好”、“今天天气怎么样？”（如果 LLM 有能力直接回答或通过简单工具查询）。
    *   **需要工具/复杂规划**：用户意图需要多步骤协作、调用外部工具或进行复杂逻辑处理才能完成。例如：“帮我调研一下 AI 领域的最新进展并生成一份报告”。
    *   **模棱两可/需要澄清**：用户输入不明确，无法直接判断意图，需要进一步向用户提问以获取更多信息。

### 2.2. 快速回答模式 (Fast Path)

*   **触发**：如果分类器判断用户意图属于“闲聊/问候/简单问答”。
*   **执行**：直接将用户输入提交给 LLM 进行回复，不经过 Planner 和 Executor。
*   **优势**：
    *   **响应速度快**：省去了规划和执行的开销。
    *   **Token 消耗极低**：只进行一次 LLM 调用。
    *   **用户体验好**：对于简单请求，能迅速给出答案，避免不必要的等待。

### 2.3. 规划模式 (Planning Path)

*   **触发**：如果分类器判断用户意图属于“需要工具/复杂规划”。
*   **执行**：启动完整的 Planner -> Executor 流程，按照之前讨论的设计进行任务规划和执行。

### 2.4. 用户干预/切换机制

*   **重要性**：即使智能分类器再智能，也可能出现误判。因此，提供用户手动干预和切换模式的能力至关重要。
*   **示例**：
    *   **纠正快速模式**：用户说“你好”，系统自动进入快速模式回复“你好！有什么可以帮你的吗？”。用户可以接着说：“不，我需要你进入规划模式，帮我分析一下市场趋势。”
    *   **强制快速模式**：用户说“帮我查一下某个股票价格”，系统可能默认进入规划模式（因为它需要调用工具）。用户可以说：“不用规划，直接告诉我现在的价格就行。”（如果系统能快速调用一个简单的工具）。

**总结**：

*   **自主判断**是实现无缝用户体验的首选机制。
*   **快速回答模式**是提升效率和节约成本的核心策略。
*   **规划模式**用于处理需要复杂逻辑和工具协作的任务。
*   **用户可以干预和纠正模式**，作为系统的安全网和提升灵活性的关键。

这种分层处理能够很好地平衡“轻量化”与“强大功能”之间的需求。
